## 💬 預期會出現的對話

### 工程師可能說：
```
「改成 -2~2 會更快。」
↓ 你應該問：
「快多少？具體小時數？」
「後續改歐氏還要多久？」
「總工時會不會其實一樣？」
```

### PM 可能說：
```
「工期真的很緊。」
↓ 你應該問：
「緊到什麼程度？精確到小時？」
「什麼功能會被延期？」
「能否協商優先級？」
```

### 你的回應範本：
```
「我理解時間壓力。讓我們看看數據...」
（拉出對比表）

「方案 A 節省的時間是？」
「方案 B 延期的時間是？」
「淨差異是多少？」

「而且方案 B 解決了根本問題，
 方案 A 只是延後問題。」
```

### 時間成本（重要數據）
```

方案 B：從一開始就完善
```

```
方案 A：埋下中-高技術債

```

## 🚩 可能的卡點

### 卡點 1：「工期壓力」
**應對：** 要求具體數字，而不是模糊的「很緊」

### 卡點 2：「工程師說改不了」

### 卡點 4：「不想改」
**應對：** 回到數據和用戶體驗論點

## 🎬 最後的心理準備

### 記住
- 你的立場是**合理的**（算法質量很重要）
- 時間數據**支持你**（改算法其實不省時間）
- 用戶體驗**要優先**（這是 MVP 的核心）

### 態度
- 👍 開放傾聽
- 👍 尊重工程師的技術考量
- 👍 理解 PM 的工期壓力
- 👍 但堅持基於數據的論點
